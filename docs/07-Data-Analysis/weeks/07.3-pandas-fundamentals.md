## ЁЯза Mental model

> **A DataFrame is a dict of columns. Each column is a NumPy array. They share one Index.**

```mermaid
flowchart TB
    subgraph DF["DataFrame"]
        I["Index<br/>['a','b','c']<br/>(shared by all columns)"]
        C1["'name' тЖТ object array"]
        C2["'age' тЖТ int64 array"]
        C3["'score' тЖТ float64 array"]
    end
    C1 & C2 & C3 -.->|"aligned by"| I
    style I fill:#e67e22,color:#fff
    style C2 fill:#4a90d9,color:#fff
```

**Three consequences you must internalize:**

1. **Columns are fast; rows are slow.** A column is one contiguous NumPy array. A *row* cuts across every column тАФ different dtypes, different arrays. **This is why `df.iterrows()` is 100├Ч slower than a vectorized operation**, and why the entire Pandas idiom is column-oriented.
2. **The Index is not decoration тАФ it's an alignment key.** Every operation between two Pandas objects **aligns on the index first**. This is Pandas' superpower and its most confusing behaviour.
3. **A `float64` column and an `object` column behave completely differently.** Dtype is not cosmetic; it determines speed, memory, and whether your `.mean()` even works.

## ЁЯУЦ Core theory

### Series тАФ a labeled 1-D array

```python
import pandas as pd
import numpy as np

s = pd.Series([10, 20, 30], index=['a', 'b', 'c'], name='score')
print(s.values)     # [10 20 30]   тЖР it's a NumPy array underneath
print(s.index)      # Index(['a','b','c'])
print(s.dtype)      # int64
```

**Automatic alignment тАФ the behaviour that surprises everyone:**

```python
a = pd.Series([1, 2, 3], index=['x', 'y', 'z'])
b = pd.Series([10, 20, 30], index=['z', 'y', 'x'])    # REVERSED order

print(a + b)
# x    31    тЖР 1 + 30   (aligned on the LABEL, not the position!)
# y    22
# z    13
```

> [!IMPORTANT]
> **Pandas aligns on the index, not on position.** This is a feature тАФ it prevents you from accidentally adding January's revenue to February's costs. But it means a mismatched index produces `NaN` instead of an error:
> ```python
> a = pd.Series([1,2,3], index=['x','y','z'])
> c = pd.Series([1,2,3], index=[0,1,2])
> print(a + c)     # x NaN, y NaN, z NaN, 0 NaN, 1 NaN, 2 NaN  ЁЯШ▒ six NaNs, no error
> ```
> **If your arithmetic suddenly produces all-NaN, check that your indexes match.** This is one of the top three Pandas confusions, and it costs everyone an afternoon exactly once.

### DataFrame тАФ a dict of Series

```python
df = pd.DataFrame({
    'user_id':  [1, 2, 3, 4],
    'name':     ['Ada', 'Alan', 'Grace', 'Edsger'],
    'age':      [36, 41, 45, 72],
    'score':    [99.5, 97.2, 98.8, np.nan],
    'country':  ['UK', 'UK', 'US', 'NL'],
})

df.info()          # dtypes + memory тАФ RUN THIS FIRST, ALWAYS
df.describe()      # numeric summary
df.head()
df.shape           # (4, 5)
```

> [!TIP]
> **`df.info()` is the first thing you run on any dataset, every time.** It tells you the row count, the dtypes (is `age` accidentally an object?), the non-null counts (where are the missing values?), and the memory usage тАФ **four of the five things that will go wrong, in one command.** Make it a reflex.

## ЁЯП╖я╕П The Index

The Index is a **hash table** (for label lookup) plus an array. That's why:

| Index type | Lookup | Use for |
|---|---|---|
| **RangeIndex** (default) | O(1) | Just row numbers. Costs nothing |
| **Int64/Object Index** | O(1) hash lookup | Meaningful keys (user_id) |
| **DatetimeIndex** | O(1), + time-slicing superpowers | **Time series тАФ always** |
| **MultiIndex** | Hierarchical | Grouped/panel data |

```python
df = df.set_index('user_id')       # promote a column to the index
df.loc[3]                          # O(1) lookup by label
df = df.reset_index()              # demote it back to a column
```

> [!NOTE]
> **When should you set an index?** When you'll look up by that key repeatedly (it becomes a hash lookup), when it's a **DatetimeIndex** (unlocks resampling and time-slicing, [07.4](07.4-pandas-advanced.md)), or when you're aligning two frames on it. Otherwise the default RangeIndex is fine тАФ **don't set an index just because it looks tidy.**

## ЁЯОп `.loc` vs `.iloc` тАФ get this right once

| | `.loc` | `.iloc` |
|---|---|---|
| Indexes by | **Label** | **Integer position** |
| Slice endpoint | **INCLUSIVE** тЪая╕П | exclusive (like Python) |
| Boolean mask | тЬЕ | тЭМ (needs `.values`) |

```python
df = pd.DataFrame({'a': [1,2,3,4]}, index=['w','x','y','z'])

df.loc['x']            # by label
df.loc['x':'z']        # тЪая╕П INCLUSIVE тАФ returns x, y, AND z (3 rows)
df.iloc[1]             # by position
df.iloc[1:3]           # exclusive тАФ returns x, y (2 rows)

df.loc[df['a'] > 2]                # boolean mask тЬЕ
df.loc[df['a'] > 2, 'a'] = 0       # тЬЕ mask + column, in ONE call тАФ the correct way
```

> [!CAUTION]
> **`.loc` slicing is INCLUSIVE of the endpoint.** `df.loc['2024-01-01':'2024-01-31']` includes January 31st. `df.iloc[0:31]` does not include row 31. This asymmetry is deliberate (label slicing is more natural when inclusive) and it catches everyone at least once.

## ЁЯЪи SettingWithCopyWarning тАФ the Pandas rite of passage

**This is the most misunderstood warning in Python, and it's a real bug, not noise.**

```python
# тЭМ CHAINED INDEXING тАФ this may or may not work. You cannot tell.
df[df['age'] > 40]['score'] = 100
# SettingWithCopyWarning: A value is trying to be set on a copy of a slice...
# ...and the original df is UNCHANGED. Your code did nothing.

# тЬЕ ONE .loc call тАФ always works
df.loc[df['age'] > 40, 'score'] = 100
```

**Why does it happen?** `df[mask]` returns a **new object**, which may be a view or a copy of the underlying NumPy buffer ([07.2](07.2-numpy.md)) тАФ **Pandas cannot always tell which.** So `df[mask]['col'] = x` assigns to a temporary that may be immediately discarded. **Sometimes it works, sometimes it silently doesn't.** That non-determinism is exactly why it warns.

> [!IMPORTANT]
> **The rule: never chain a `[...]` with an assignment. Use exactly one `.loc[rows, cols] = value`.**
>
> And when you want an independent subset, **say so**: `subset = df[df.age > 40].copy()`. The `.copy()` is not paranoia; it's the difference between code that works and code that works *on Tuesdays*.
>
> (Pandas 3.0's Copy-on-Write makes this deterministic тАФ subsets always behave as copies. But the `.loc` habit is correct in every version, so build it now.)

## ЁЯТ╛ Dtypes тАФ the 10├Ч memory win nobody uses

```python
import pandas as pd, numpy as np

n = 1_000_000
df = pd.DataFrame({
    'id':      np.arange(n),                                  # int64  тЖТ 8 MB
    'country': np.random.choice(['US','UK','DE','FR'], n),    # object тЖТ 60+ MB!
    'active':  np.random.choice([True, False], n),            # bool   тЖТ 1 MB
    'score':   np.random.rand(n),                             # float64тЖТ 8 MB
})
print(f"before: {df.memory_usage(deep=True).sum()/1e6:.1f} MB")   # ~70 MB

df['country'] = df['country'].astype('category')   # тЖР THE BIG ONE
df['id']      = df['id'].astype('int32')
df['score']   = df['score'].astype('float32')
print(f"after : {df.memory_usage(deep=True).sum()/1e6:.1f} MB")   # ~8 MB тЖТ 9├Ч smaller
```

> [!IMPORTANT]
> **`category` is the single biggest memory win in Pandas, and almost nobody uses it.** An `object` column stores a **pointer to a separate Python string object for every row** тАФ a million rows with the value `"US"` stores a million pointers. A `category` stores the four unique values *once* plus a compact integer code array. **On low-cardinality string columns (country, status, product_type, gender), it routinely cuts memory by 10тАУ50├Ч and makes groupby dramatically faster.**
>
> **The rule of thumb: if a string column has fewer than ~50% unique values, make it a category.**

| Dtype | Bytes/row | Use for |
|---|---|---|
| `int8` / `int16` / `int32` | 1 / 2 / 4 | **Check your actual range** тАФ ages fit in int8 |
| `float32` | 4 | Almost everything numeric. float64 is overkill |
| `bool` | 1 | Flags |
| **`category`** | ~1 + a small dict | **Low-cardinality strings** тЖР the big win |
| `object` | 8 (ptr) + ~50 (str) | Genuinely free text only |
| `datetime64[ns]` | 8 | Always parse dates тАФ never leave them as strings |
| `Int64` (capital I) | 8 + mask | Integers **with missing values** (nullable) |

> [!NOTE]
> **Why is `int64` with a NaN silently upcast to `float64`?** Because NumPy's `int64` has **no representation for missing**. `NaN` is a float concept. So the moment one value is missing, the whole column becomes float тАФ and your IDs become `1.0, 2.0, 3.0`. **Pandas' nullable `Int64` (capital I) fixes this** with a separate boolean mask. Use it when integer columns can be null.

## ЁЯУе Reading & writing data

```python
import pandas as pd

# тФАтФА CSV тАФ universal, and the wrong default тФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФА
df = pd.read_csv(
    'data.csv',
    dtype={'user_id': 'int32', 'country': 'category'},   # тЖР set dtypes AT READ TIME
    parse_dates=['created_at'],                          # тЖР parse dates AT READ TIME
    usecols=['user_id', 'country', 'created_at', 'amount'],  # тЖР read only what you need
    na_values=['', 'NULL', 'N/A', '-1', 'unknown'],      # тЖР declare your sentinels!
)

# тФАтФА Parquet тАФ what you should actually use тФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФАтФА
df.to_parquet('data.parquet', compression='snappy')
df = pd.read_parquet('data.parquet', columns=['user_id', 'amount'])  # column pruning!
```

| Format | Size | Read speed | Types preserved? | Use when |
|---|---|---|---|---|
| **CSV** | 1├Ч | 1├Ч | тЭМ **No** тАФ everything is a string | Interop with humans/Excel |
| **Parquet** | **~0.2├Ч** | **~10├Ч** | тЬЕ Yes | тЬЕ **Your default for anything internal** |
| Feather/Arrow | ~0.3├Ч | **~30├Ч** | тЬЕ Yes | Fast local cache, IPC |
| Pickle | ~0.5├Ч | fast | тЬЕ Yes | тЪая╕П **Insecure + version-fragile.** Avoid |
| JSON | 2├Ч | slow | partial | APIs, nested data |

> [!IMPORTANT]
> **CSV has no schema.** Every read is a *guess*: is `007` a string or the number 7? Is `2024-01-02` a date or text? Is the empty string a missing value or a legitimate empty? Pandas guesses тАФ and it guesses differently depending on what's in the first few thousand rows, which means **the same code can behave differently on different files.**
>
> **Parquet is columnar, compressed, typed, and ~10├Ч faster** тАФ and column pruning means reading 3 columns out of 200 costs almost nothing. **Use CSV to talk to humans; use Parquet for everything else.** ([07.10](07.10-performance.md))

> [!CAUTION]
> **`na_values` is the one you'll forget.** Real data encodes missing as `-1`, `999`, `"N/A"`, `"unknown"`, `"NULL"`, or an empty string. **If you don't declare them, `-1` becomes a legitimate age and poisons every statistic you compute.** This is exactly the silent data bug from [07.1](07.1-data-lifecycle.md): `df['age'].mean()` returns 43.7 and it's a lie.

## ЁЯЧВя╕П MultiIndex

A hierarchical index тАФ multiple levels per row.

```python
import pandas as pd, numpy as np

df = pd.DataFrame({
    'country': ['US','US','UK','UK'],
    'year':    [2023, 2024, 2023, 2024],
    'revenue': [100, 120, 80, 95],
}).set_index(['country', 'year'])       # тЖР two levels

print(df.loc['US'])                     # all US rows
print(df.loc[('US', 2024)])             # one specific cell тАФ a TUPLE
print(df.xs(2024, level='year'))        # cross-section: all countries, 2024
print(df.unstack('year'))               # pivot year into columns
```

> [!TIP]
> **MultiIndexes arrive whether you want them or not** тАФ `groupby(['a','b']).sum()` produces one automatically. Two survival commands: **`.reset_index()`** flattens it back to plain columns (usually what you want), and **`.unstack()`** pivots a level into columns. If a MultiIndex is confusing you, `reset_index()` and move on. **Elegance is not worth an hour of confusion.**

## тЪб Performance considerations

### The iteration ladder тАФ memorize this ranking

```python
import pandas as pd, numpy as np, time
df = pd.DataFrame({'a': np.random.rand(100_000), 'b': np.random.rand(100_000)})

# тЭМтЭМтЭМ WORST тАФ iterrows: ~5.0 s. Creates a Series per row.
result = [row['a'] * row['b'] for _, row in df.iterrows()]

# тЭМтЭМ BAD тАФ apply with axis=1: ~1.2 s. Still a Python call per row.
result = df.apply(lambda r: r['a'] * r['b'], axis=1)

# тЭМ MEH тАФ itertuples: ~0.08 s. Better, but still Python-level.
result = [r.a * r.b for r in df.itertuples()]

# тЬЕ GOOD тАФ vectorized: ~0.0005 s
result = df['a'] * df['b']
```

**`iterrows` тЖТ vectorized is a 10,000├Ч speedup.** Not a typo.

| Method | Relative time | Why |
|---|---|---|
| `.iterrows()` | **10,000├Ч** | Constructs a **new Series object for every row** |
| `.apply(axis=1)` | 2,400├Ч | A Python function call per row |
| `.itertuples()` | 160├Ч | Namedtuples тАФ lighter, still Python |
| `.apply()` on a **column** | ~50├Ч | Python call per element |
| **Vectorized** | **1├Ч** | тЬЕ One C-level operation on the whole array |
| `.map()` with a dict | ~2├Ч | Fine for lookups |

> [!WARNING]
> **`df.iterrows()` is the single most damaging habit in Pandas.** It appears in every tutorial and it is almost always wrong. It's slow because it **constructs a brand-new Series object for every single row** тАФ which also means it *loses dtypes* (a row with mixed types becomes all-object).
>
> **If you find yourself reaching for `iterrows`, stop and ask: what is the column-wise version of this?** There nearly always is one тАФ `np.where`, `.map`, a merge, a groupby, or plain arithmetic.

### Vectorized string operations

```python
df['email'].str.lower().str.strip()
df['name'].str.contains('smith', case=False, na=False)   # тЖР na=False matters!
df['phone'].str.replace(r'\D', '', regex=True)
df['full'] = df['first'] + ' ' + df['last']
```

`.str` is vectorized тАФ still Python-level under the hood (strings are objects), but ~10├Ч faster than `apply`, and infinitely more readable.

### Other performance rules

```python
# тЭМ O(n┬▓) тАФ every concat COPIES the whole frame
result = pd.DataFrame()
for f in files:
    result = pd.concat([result, pd.read_csv(f)])     # quadratic!

# тЬЕ O(n) тАФ collect, concat once
result = pd.concat([pd.read_csv(f) for f in files], ignore_index=True)
```

**Same lesson as `np.append`** ([07.2](07.2-numpy.md)): growing an immutable structure in a loop is quadratic. Build a list; concatenate once.

## ЁЯФТ Security & privacy considerations

| Concern | Note |
|---|---|
| **`pd.read_pickle` on untrusted data** | ЁЯТА **Arbitrary code execution.** Pickle can execute anything on load. **Never** unpickle a file you didn't create |
| **`df.head()` in a committed notebook** | Publishes **real customer records** into git history, permanently. Use `nbstripout` as a [pre-commit hook](../../04-Git/weeks/04.10-hooks.md) |
| **`df.to_csv()` writes everything** | Including PII columns you forgot were there. **Whitelist columns explicitly** on export, never blacklist |
| **`read_csv` from a URL** | Fetches and parses arbitrary remote content. Validate the source and the schema |
| **PII in the index** | Easy to forget it's there тАФ it survives `df.drop(columns=[...])` |
| **`df.query()` / `eval()`** | Uses `eval` semantics. **Never** interpolate user input into a query string |
| **Memory dumps / cached parquet** | Intermediate files in `/tmp` or `data/` outlive your process and often escape access controls |

```python
# тЭМ blacklist тАФ you WILL forget a column
df.drop(columns=['email', 'ssn']).to_csv('export.csv')

# тЬЕ whitelist тАФ safe by construction
SAFE = ['user_id_hashed', 'country', 'signup_month', 'plan_tier']
df[SAFE].to_csv('export.csv', index=False)
```

**Whitelist, never blacklist.** A new PII column added upstream silently appears in a blacklisted export. It cannot appear in a whitelisted one.

## тЬЕ Best practices

| Practice | Why |
|---|---|
| **`df.info()` first, always** | Rows, dtypes, nulls, memory тАФ four bugs in one command |
| **Set dtypes at read time** | Not after. Saves memory *and* prevents wrong inference |
| **Declare `na_values`** | `-1`, `999`, `"N/A"` are missing values pretending to be data |
| **`category` for low-cardinality strings** | 10тАУ50├Ч memory reduction |
| **`parse_dates` at read time** | A date left as a string is a bug waiting to happen |
| **One `.loc[rows, cols] = val`** | Never chain `[...]` with assignment |
| **`.copy()` for independent subsets** | Explicit beats "works on Tuesdays" |
| **Parquet, not CSV** | Typed, columnar, 5├Ч smaller, 10├Ч faster |
| **Never `iterrows`** | Vectorize. It's 10,000├Ч |
| **`ignore_index=True`** on concat | Otherwise you get duplicate index values, which break `.loc` |

## ЁЯРЫ Common mistakes

| Mistake | Symptom | Fix |
|---|---|---|
| **`df[mask]['col'] = x`** | `SettingWithCopyWarning`; the assignment **silently does nothing** | `df.loc[mask, 'col'] = x` |
| **`df.iterrows()`** | 10,000├Ч too slow; loses dtypes | Vectorize |
| Not declaring `na_values` | `-1` becomes a valid age; every stat is wrong | `na_values=[-1, 'N/A', ...]` |
| Leaving strings as `object` | 10тАУ50├Ч memory waste | `.astype('category')` |
| Mismatched indexes in arithmetic | **All NaN**, no error | Check `.index`; `reset_index()` |
| Forgetting `.loc` slicing is **inclusive** | Off-by-one on date ranges | It's deliberate. Remember it |
| `pd.concat` in a loop | O(n┬▓) | Build a list, concat once |
| `int64` column silently becomes `float64` | IDs become `1.0, 2.0` | A NaN appeared. Use nullable `Int64` |
| `.str.contains()` with NaN | `ValueError` | `na=False` |
| `pd.read_pickle` on untrusted data | **RCE** | Never |
| `inplace=True` | Doesn't save memory (a myth), hurts chaining, being deprecated | Just reassign: `df = df.drop(...)` |

**Implementation guidance**
1. **Read with `dtype=str` first.** Let *your* code decide the types, not Pandas' guesswork тАФ that's the whole point of a profiler.
2. **`sentinels.py` is the valuable part.** Look for suspicious values: negative ages, `999`/`-999`, string forms of null, and **values that are far more frequent than their neighbours** (a spike at exactly `-1` in an otherwise smooth distribution is a sentinel, not data). **This module catches the exact silent bug that produced the wrong `mean()` in [07.1](07.1-data-lifecycle.md).**
3. **`dtypes.py`** тАФ for each column, compute the actual min/max and pick the smallest int type that fits; suggest `category` when uniqueness < 50%. Report total memory saved.

**Testing strategy**
- Build `examples/messy.csv` **deliberately**: a `-1` age, a `999` sentinel, a constant column, a country column with 3 unique values in 10,000 rows, a date column in mixed formats, and an integer ID column with one null.
- Assert the profiler finds **every planted problem**. This is a form of mutation testing тАФ you know the ground truth because you wrote the bugs.
- `test_dtype_suggestion`: assert the country column is recommended as `category` and that the reported saving is > 90%.

**Future improvements**
- Emit a `pandera` schema from the profile тАФ turning a one-off analysis into a **reusable validation contract** ([07.9](07.9-data-quality.md)).
- Compare two profiles to detect **drift** between last month's file and this month's.
- Add a `--fix` flag that writes a cleaned Parquet with the suggested dtypes applied.

**Why this project:** because **you will point it at every new dataset you ever receive**, and it will save you from the class of bug that has no stack trace.

## ЁЯО┤ Flashcards

- **Q:** What is a DataFrame internally? тЖТ **A:** A **dict of columns**, each a NumPy array, sharing one Index. Hence columns are fast and rows are slow.
- **Q:** Why is `df.iterrows()` so slow? тЖТ **A:** It **constructs a new Series object for every row** (and loses dtypes). It's ~10,000├Ч slower than vectorized.
- **Q:** What causes `SettingWithCopyWarning`? тЖТ **A:** **Chained indexing** тАФ `df[mask]['col'] = x` assigns to a temporary that may be a copy, so the assignment **may silently do nothing**. Fix: one `df.loc[mask, 'col'] = x`.
- **Q:** `.loc` vs `.iloc`? тЖТ **A:** `.loc` is by **label** and its slice is **INCLUSIVE**; `.iloc` is by **position** and exclusive.
- **Q:** Why does an int column become float when a value goes missing? тЖТ **A:** NumPy's `int64` has **no NaN representation**. Use the nullable `Int64` (capital I) instead.
- **Q:** Why is `category` such a big memory win? тЖТ **A:** An `object` column stores a **pointer to a separate Python string per row**. A `category` stores each unique value **once** plus a compact integer code array. 10тАУ50├Ч on low-cardinality strings.
- **Q:** When should you use `category`? тЖТ **A:** When a string column has **< ~50% unique values** (country, status, plan_tier).
- **Q:** Why does Pandas arithmetic between two Series produce NaN? тЖТ **A:** **Index misalignment.** Pandas aligns on **labels**, not position тАФ mismatched labels produce NaN rather than an error.
- **Q:** Why is Parquet better than CSV? тЖТ **A:** Typed (no guessing), columnar (read only the columns you need), compressed (~5├Ч smaller), and ~10├Ч faster. **CSV has no schema.**
- **Q:** Why does `na_values` matter so much? тЖТ **A:** Real data encodes missing as `-1`, `999`, `"N/A"`. Undeclared, they become **legitimate values** and silently poison every statistic.
- **Q:** Why is `pd.read_pickle` on untrusted data dangerous? тЖТ **A:** Pickle can execute **arbitrary code** on load.
- **Q:** Whitelist or blacklist on export? тЖТ **A:** **Whitelist.** A new PII column added upstream silently appears in a blacklisted export; it cannot appear in a whitelisted one.

## ЁЯТ╝ Interview questions

1. **"Your Pandas job takes 3 hours. Where do you look first?"** тАФ `iterrows`/`apply(axis=1)` (vectorize), `concat` in a loop (O(n┬▓)), object dtypes (categorize), reading CSV instead of Parquet, and reading columns you don't need. **In that order** тАФ it's roughly the order of impact.
2. **"Explain `SettingWithCopyWarning`."** тАФ Chained indexing produces an ambiguous view-or-copy, so the assignment may silently do nothing. Fix with a single `.loc`. **Emphasize that it's a real bug, not noise** тАФ most candidates suppress it.
3. **"How would you cut a DataFrame's memory by 10├Ч?"** тАФ `category` for low-cardinality strings (the big one), downcast ints/floats, drop unused columns at read time, and use Parquet with column pruning.
4. **"`.loc` vs `.iloc`?"** тАФ Label vs position; and mention that **`.loc` slicing is inclusive**. That detail is the tell that you've actually used it.
5. **"Your `df['age'].mean()` returns 43.7 but you suspect it's wrong. What do you check?"** тАФ Missing-value **sentinels** (`-1`, `999`), the dtype (is it object?), nulls silently excluded, and outliers. **Plot the distribution** тАФ a spike at exactly `-1` is not data.

## ЁЯУЪ Summary

- **A DataFrame is a dict of NumPy arrays sharing an Index.** Columns are contiguous and fast; rows cut across dtypes and are slow. **The entire Pandas idiom follows from this.**
- **Pandas aligns on the index, not position.** Mismatched indexes give you **NaN, not an error** тАФ check `.index` when arithmetic mysteriously produces all-NaN.
- **`.loc` is by label (slice INCLUSIVE); `.iloc` is by position (exclusive).**
- **Never chain `[...]` with assignment.** `df[mask]['col'] = x` may silently do nothing. **One `.loc[rows, cols] = value`.**
- **Never use `.iterrows()`** тАФ 10,000├Ч slower than vectorized, and it loses dtypes.
- **`category` is the biggest memory win in Pandas** тАФ 10тАУ50├Ч on low-cardinality string columns, and nobody uses it.
- **Set dtypes, parse dates, and declare `na_values` AT READ TIME.** Undeclared sentinels like `-1` become legitimate data and silently poison every statistic тАФ the exact silent bug from [07.1](07.1-data-lifecycle.md).
- **CSV has no schema; Parquet does.** Typed, columnar, 5├Ч smaller, 10├Ч faster. **Use CSV to talk to humans; Parquet for everything else.**
- **`df.info()` first, every time.** And **whitelist columns on export** тАФ never blacklist.
