## 3. Why This Topic Exists

AI Engineers automate constantly: "set up the environment, download the dataset, run training, back up the model, restart the service." Doing this by hand is slow and error-prone; a bash script does it reliably, repeatably, and can run unattended (cron, CI, [Module 16](../../16-MLOps/README.md)). Bash is the glue of Linux automation — deployment scripts, entrypoints for Docker containers ([03.16](03.16-docker-preparation.md)), and setup scripts are all bash.

The catch: bash is *easy to write badly*. A script without error handling silently continues after a failure, potentially deleting the wrong files or deploying a broken model. This lesson emphasizes **robust** scripting — the difference between automation you can trust and automation that causes 3 a.m. incidents.

> [!IMPORTANT]
> The single most important bash habit: **`set -euo pipefail` at the top of every script.** By default, bash *ignores errors and continues* — a failed `cd` followed by `rm -rf *` deletes the wrong directory. `set -e` (exit on error), `-u` (error on undefined variables), and `-o pipefail` (catch failures in pipelines) make scripts *fail loudly and safely* instead of silently corrupting things. Never write a non-trivial script without it.

## 4. A First Robust Script

```bash
#!/usr/bin/env bash          # shebang: run with bash
set -euo pipefail            # exit on error, undefined var, or pipe failure

DATA_DIR="/data/dataset"     # variables (no spaces around =)
readonly DATA_DIR            # make it constant

echo "Preparing ${DATA_DIR}..."
mkdir -p "${DATA_DIR}"       # quote variables to handle spaces!
echo "Done."
```

```bash
chmod +x setup.sh            # make it executable (03.6)
./setup.sh                   # run it
```

| Element | Purpose |
|---|---|
| `#!/usr/bin/env bash` | **Shebang** — tells the OS to run this with bash |
| `set -euo pipefail` | Fail fast on errors/undefined vars/pipe failures |
| `VAR="value"` | Variable (no spaces around `=`) |
| `"${VAR}"` | Use a variable — **always quote it** |

> [!WARNING]
> **Always quote your variables: `"${VAR}"`.** An unquoted `$VAR` containing spaces or being empty causes bugs and disasters — `rm -rf $DIR/` where `$DIR` is empty becomes `rm -rf /`. Quoting (`"${DIR}"`) and `set -u` (error on unset variables) together prevent the catastrophic empty-variable class of bugs ([03.5](03.5-essential-commands.md)). Use `"${DIR:?msg}"` to hard-fail if a variable is unset.

## 5. Variables and Parameters

| Form | Meaning |
|---|---|
| `VAR="value"` | Assign (no spaces) |
| `"${VAR}"` | Reference (quoted, braced) |
| `$1`, `$2`, … | Positional parameters (script arguments) |
| `$0` | The script name |
| `$#` | Number of arguments |
| `$@` | All arguments |
| `$(command)` | Command substitution (capture output) |
| `${VAR:-default}` | Use `default` if `VAR` is unset |

```bash
#!/usr/bin/env bash
set -euo pipefail

MODEL_NAME="${1:?Usage: $0 <model-name>}"   # required arg, or fail with usage
EPOCHS="${2:-10}"                            # optional, default 10
TIMESTAMP="$(date +%Y%m%d_%H%M%S)"           # command substitution

echo "Training ${MODEL_NAME} for ${EPOCHS} epochs (run ${TIMESTAMP})"
```

> [!TIP]
> `${1:?Usage...}` gives you free argument validation — the script exits with a usage message if the argument is missing. `${2:-10}` provides defaults. `$(command)` captures output into a variable (`FILES="$(ls *.csv)"`). These make scripts robust and self-documenting.

## 6. Conditionals

```bash
if [[ -d "${DATA_DIR}" ]]; then
    echo "Dataset exists."
elif [[ -f "${DATA_DIR}.tar.gz" ]]; then
    echo "Archive exists; extracting..."
    tar xzf "${DATA_DIR}.tar.gz"
else
    echo "No dataset found." >&2      # error to stderr (03.4)
    exit 1                             # exit with error code
fi
```

| Test | True when |
|---|---|
| `[[ -f FILE ]]` | FILE exists and is a regular file |
| `[[ -d DIR ]]` | DIR exists and is a directory |
| `[[ -z "$X" ]]` / `[[ -n "$X" ]]` | X is empty / non-empty |
| `[[ "$A" == "$B" ]]` | Strings equal |
| `[[ "$A" -eq "$B" ]]` | Numbers equal (`-gt`, `-lt`, …) |
| `command; if [[ $? -eq 0 ]]` | Previous command succeeded |

> [!TIP]
> Use **`[[ ]]`** (bash's test) over the older `[ ]` — it's safer with strings/variables and supports `&&`/`||`. Send error messages to **stderr** (`>&2`, [03.4](03.4-terminal-mastery.md)) so they're separable from normal output, and `exit 1` (non-zero) on failure so callers/CI know it failed (§9).

## 7. Loops

```bash
# Loop over files (glob):
for file in /data/*.csv; do
    echo "Processing ${file}..."
    python process.py "${file}"
done

# Loop over a list:
for gpu in 0 1 2 3; do
    CUDA_VISIBLE_DEVICES="${gpu}" python train.py --shard "${gpu}" &   # parallel!
done
wait                                   # wait for all background jobs (03.7)

# While loop (read a file line by line):
while IFS= read -r line; do
    echo "Line: ${line}"
done < input.txt
```

| Loop | Use |
|---|---|
| `for x in a b c` | Iterate a list/glob |
| `for f in *.csv` | Iterate matching files (careful with globs, [03.4](03.4-terminal-mastery.md)) |
| `while read -r line; do ...; done < file` | Process a file line by line |
| `until [[ condition ]]` | Loop until true |

> [!IMPORTANT]
> Loops in bash are for **orchestration**, not heavy computation (recall [Module 02.1](../../02-Computer-Science/weeks/02.1-how-computers-work.md): bash is slow for number-crunching). The AI pattern above — launching a training shard per GPU in the background (`&`) and `wait`-ing — is real, useful bash. For *data processing*, prefer the composed tools ([03.5](03.5-essential-commands.md)) or Python/pandas ([Module 07](../../07-Data-Analysis/README.md)); for *orchestrating commands*, bash loops are perfect. `while IFS= read -r line` is the correct, safe idiom for reading files line-by-line (preserves whitespace, handles backslashes).

## 8. Functions

```bash
#!/usr/bin/env bash
set -euo pipefail

log() {                                # a reusable logging function
    echo "[$(date +%H:%M:%S)] $*"      # $* = all arguments
}

backup_model() {
    local src="$1"                     # 'local' scopes the variable to the function
    local dest="$2"
    log "Backing up ${src} → ${dest}"
    rsync -a "${src}" "${dest}"        # (03.9)
}

log "Starting backup"
backup_model /data/models/v3 /backup/
log "Done"
```

> [!TIP]
> Use **functions** to avoid repetition and structure larger scripts, and always declare function variables with **`local`** so they don't leak into the global scope (a subtle bug source). A `log()` helper (timestamped output to a file/stderr) is a great habit — it's your script's version of [Module 01.9's](../../01-Advanced-Python/weeks/01.9-error-handling-logging.md) logging.

## 9. Exit Codes and Error Handling

Every command returns an **exit code**: `0` = success, non-zero = failure ([Module 02.6](../../02-Computer-Science/weeks/02.6-operating-systems.md) — e.g., 137 = OOM-killed). This is how scripts, CI, and systemd know if something worked.

```bash
#!/usr/bin/env bash
set -euo pipefail

# Check a command's success explicitly:
if ! python train.py; then
    echo "Training failed!" >&2
    exit 1
fi

# Cleanup on exit (runs even on error — like a context manager, Module 01.7):
cleanup() {
    echo "Cleaning up temp files..."
    rm -f /tmp/run_*.tmp
}
trap cleanup EXIT                      # run cleanup() whenever the script exits

# ${?} holds the last exit code
some_command
echo "Exit code was: ${?}"
```

| Mechanism | Purpose |
|---|---|
| `$?` | Exit code of the last command (0 = success) |
| `exit N` | End the script with code N (0 success, non-zero failure) |
| `set -e` | Exit immediately if any command fails |
| `trap cleanup EXIT` | Run a cleanup function on exit (even on error) |
| `command \|\| { echo fail; exit 1; }` | Handle a specific command's failure |

> [!IMPORTANT]
> **Exit codes are the contract between your script and everything that runs it.** systemd's `Restart=on-failure` ([03.8](03.8-services-systemd.md)), CI pipelines ([Module 16](../../16-MLOps/README.md)), and callers all check the exit code — so *always `exit 1` on failure and `exit 0` on success*. `trap cleanup EXIT` is the bash equivalent of a context manager ([Module 01.7](../../01-Advanced-Python/weeks/01.7-context-managers.md)) — it guarantees cleanup (temp files, released resources) runs even if the script errors partway. Robust error handling here is the same discipline as [Module 01.9](../../01-Advanced-Python/weeks/01.9-error-handling-logging.md).

## 10. Bash vs Python — When to Use Which

| Use bash when… | Use Python when… |
|---|---|
| Orchestrating commands/tools | Real logic, data structures, math |
| Gluing a pipeline together | Parsing/transforming complex data |
| Short setup/deploy/entrypoint scripts | Anything with error handling depth, testing |
| It's < ~50 lines | It's growing complex |

> [!TIP]
> **The rule of thumb: bash for *orchestrating* commands, Python for *logic*.** A deploy script that runs a sequence of commands → bash. A script that parses JSON, does math, or has nontrivial branching → Python (which you can test, [Module 01.10](../../01-Advanced-Python/weeks/01.10-testing.md)). When a bash script exceeds ~50 lines or grows real logic, rewrite it in Python — bash's error handling and data structures are too weak for complex programs. Many AI teams use bash for the thin "run these steps" layer and Python for everything substantive.

> [!TIP]
> Run **`shellcheck`** on your scripts — it's a linter ([Module 01.13](../../01-Advanced-Python/weeks/01.13-packaging-code-quality.md)) that catches quoting bugs, unset variables, and common mistakes *before* they bite. `shellcheck deploy.sh` will flag exactly the errors this lesson warns about. Treat it like `ruff`/`mypy` for bash.

## 11. Common Mistakes & Debugging

| Mistake | Consequence | Fix |
|---|---|---|
| No `set -euo pipefail` | Silent failures, continues after errors | Add it to every script |
| Unquoted variables | Word-splitting/empty-var disasters | Always `"${VAR}"` |
| `rm -rf $VAR/` with empty VAR | Deletes wrong/everything | `set -u`; `"${VAR:?}"` |
| Ignoring exit codes | CI/systemd think it succeeded | `exit 1` on failure |
| No cleanup on error | Leaked temp files/resources | `trap cleanup EXIT` |
| Complex logic in bash | Unmaintainable, buggy | Rewrite in Python |
| Not linting | Subtle bugs ship | `shellcheck` |
| `[ ]` with unquoted vars | Test errors | Use `[[ ]]` + quotes |

> [!TIP]
> Debug scripts with `bash -x script.sh` (prints each command as it runs — trace execution) and `set -x`/`set +x` to trace a section. This is the bash version of the [Module 02.12](../../02-Computer-Science/weeks/02.12-debugging.md) debugging mindset.

## 12. Performance Considerations

| Principle | Takeaway |
|---|---|
| Bash is slow for computation | Orchestrate, don't compute ([Module 02.1](../../02-Computer-Science/weeks/02.1-how-computers-work.md)) |
| Spawning many subprocesses | Each command is a process ([03.7](03.7-processes.md)); fine for glue, costly in tight loops |
| Parallelize with `&` + `wait` | Run independent jobs concurrently ([03.7](../../02-Computer-Science/weeks/02.8-concurrency.md)) |
| Prefer built-in text tools | `grep`/`awk` (C) beat bash loops for text ([03.5](03.5-essential-commands.md)) |

## 13. Security Considerations

| Risk | Guidance |
|---|---|
| Untrusted input in commands | Command injection — never `eval`/build commands from untrusted input ([03.4](03.4-terminal-mastery.md)) |
| Unquoted variables | Word-splitting exploits/disasters | Quote everything |
| Secrets in scripts | Don't hard-code; use env/files ([03.6](03.6-permissions.md)) |
| `curl \| bash` in scripts | Runs untrusted code ([03.6](03.6-permissions.md)) | Avoid |
| World-writable scripts | Anyone can alter them | `chmod 755`, proper owner |

> [!CAUTION]
> **Never build commands from untrusted input** in bash — `rm "$user_input"` or `eval "$user_input"` is command injection ([03.4](03.4-terminal-mastery.md)). This is especially critical if an **AI agent** generates shell commands ([Module 14](../../14-AI-Agents/README.md)): treat model-generated commands as untrusted, validate/sandbox them, and never `eval` them blindly. Hard-coded secrets in scripts are another common leak — read them from a `chmod 600` env file instead.

## 16. Cheat Sheet

```text
TEMPLATE:
  #!/usr/bin/env bash
  set -euo pipefail                    # exit on error, unset var, pipe failure — ALWAYS
  VAR="value"; readonly VAR            # no spaces around =
  echo "${VAR}"                        # ALWAYS quote & brace variables
PARAMS: $1 $2(args) · $0(name) · $#(count) · $@(all) · ${1:?usage}(required) · ${2:-default}
  $(cmd) command substitution · "${DIR:?}" hard-fail if empty (prevents rm -rf disasters)
CONDITIONALS: if [[ -f FILE ]] / [[ -d DIR ]] / [[ -z/-n "$X" ]] / [[ "$A" == "$B" ]] ; then ... fi
  errors → >&2 ; exit 1 on failure
LOOPS: for f in *.csv; do ...; done · while IFS= read -r line; do ...; done < file
  parallel: for g in 0 1 2 3; do CUDA_VISIBLE_DEVICES=$g python train.py & done; wait
FUNCTIONS: f() { local x="$1"; ...; } · log() helper
EXIT/ERRORS: $?(last code, 0=ok) · exit N · trap cleanup EXIT (guaranteed cleanup) · cmd || { echo fail; exit 1; }
BASH vs PYTHON: bash=orchestrate commands (<50 lines) · Python=logic/data/math
DEBUG: bash -x script.sh · shellcheck script.sh (LINT IT)
SECURITY: never eval/build cmds from untrusted input (injection) · no hard-coded secrets
```

## 17. Flashcards

- **Q:** What does `set -euo pipefail` do? — **A:** `-e` exit on any error, `-u` error on undefined variables, `-o pipefail` catch failures anywhere in a pipeline — makes scripts fail loudly/safely instead of silently continuing.
- **Q:** Why quote bash variables (`"${VAR}"`)? — **A:** To prevent word-splitting and empty-variable disasters (e.g., `rm -rf $DIR/` becoming `rm -rf /` when `DIR` is empty).
- **Q:** What is an exit code and why does it matter? — **A:** 0 = success, non-zero = failure; systemd, CI, and callers use it to know if the script worked — always `exit 1` on failure.
- **Q:** How do you guarantee cleanup on script exit? — **A:** `trap cleanup EXIT` runs the cleanup function whenever the script exits, even on error (bash's context-manager equivalent).
- **Q:** Bash or Python — which for what? — **A:** Bash for orchestrating commands / short glue scripts; Python for real logic, data, math, and anything you'd want to test.
- **Q:** What tool lints bash scripts? — **A:** `shellcheck` — catches quoting bugs, unset variables, and common mistakes before they bite.
