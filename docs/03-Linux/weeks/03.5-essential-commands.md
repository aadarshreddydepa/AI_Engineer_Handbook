## 3. Why This Topic Exists

These commands are your daily hands. Inspecting a dataset's first lines, tailing a training log live, finding all config files, counting rows, extracting a column, replacing a token across files — each is one command. Chained ([03.4](03.4-terminal-mastery.md)), they become ad-hoc data tools you compose in seconds, often faster than writing a Python script.

For AI Engineers specifically, `grep`/`awk`/`sed`/`find` are how you wrangle datasets, debug logs, and inspect files on a server where you *don't* have a notebook open. This lesson is a reference you'll return to for a long time.

> [!IMPORTANT]
> Don't try to memorize every flag — **learn each command's *job* and its 2–3 most-used forms**, and remember that `man <command>` (or `<command> --help`, or `tldr <command>`) has the rest. Fluency is knowing *which tool* for a task and composing them, not reciting flags. This lesson is organized by job for exactly that reason.

## 4. Navigation & Inspection

| Command | Job | Key forms |
|---|---|---|
| `pwd` | Print working directory (where am I?) | `pwd` |
| `ls` | List directory contents | `ls -la` (all + long), `ls -lh` (human sizes), `ls -lt` (by time) |
| `cd` | Change directory | `cd /path`, `cd ~`, `cd -` (previous), `cd ..` |
| `tree` | Show directory hierarchy | `tree -L 2` (2 levels deep) |

```bash
ls -lh                # long listing with human-readable sizes (K/M/G)
ls -la ~/project      # include hidden dotfiles (.env, .git)
cd -                  # jump back to the previous directory
```

> [!TIP]
> `ls -lh` (human sizes) and `ls -lt` (newest first) are the daily workhorses — the latter instantly shows the most recent checkpoint/log. `cd -` toggles between two directories (great when bouncing between code and data). `ls -la` reveals hidden dotfiles like `.env` and `.git` that a plain `ls` hides.

## 5. File Management

| Command | Job | Key forms & AI use |
|---|---|---|
| `cp` | Copy | `cp -r dir/ dest/` (recursive), `cp -a` (archive/preserve) — copy datasets/checkpoints |
| `mv` | Move / rename | `mv old new` — rename a run; move data between dirs (cheap within a filesystem, [03.3](03.3-filesystem.md)) |
| `rm` | Remove | `rm file`, `rm -r dir/`, `rm -rf dir/` — ⚠️ **irreversible** |
| `mkdir` | Make directory | `mkdir -p a/b/c` (create parents) — set up project/data trees |
| `touch` | Create empty file / update timestamp | `touch .gitkeep`, `touch file` |
| `ln` | Create links | `ln -s target link` (symlink, [03.3](03.3-filesystem.md)) |

```bash
mkdir -p data/{train,val,test}   # create a whole tree at once (brace expansion, 03.4)
cp -r /data/raw ./raw            # recursively copy a dataset
mv checkpoint.pt checkpoint_best.pt   # rename
```

> [!CAUTION]
> **`rm` is permanent — there is no Recycle Bin.** `rm -rf` recursively force-deletes with no confirmation, and `rm -rf /` (or `rm -rf $VAR/` when `$VAR` is empty!) can wipe a system. **Rules:** (1) `ls` the target first; (2) never blindly `rm -rf` a variable path without checking the variable is set (`rm -rf "${DIR:?}/"` refuses if `DIR` is empty); (3) prefer moving to a trash dir over deleting for anything important. The empty-variable `rm -rf "$DIR/*"` catastrophe has destroyed real production systems — guard against it.

## 6. Viewing File Contents

| Command | Job | When to use |
|---|---|---|
| `cat` | Print whole file (concatenate) | Small files; combining files |
| `less` | Page through a file interactively | **Large files** — doesn't load all into memory |
| `head` | First N lines (`-n 20`) | Peek at a dataset's start/header |
| `tail` | Last N lines (`-n 20`); `tail -f` follows | **Live-watch a log** as it grows |
| `wc` | Count lines/words/bytes (`-l` lines) | Count dataset rows |
| `diff` | Compare two files line by line | Compare configs/outputs |

```bash
head -5 train.csv        # inspect the header + first rows of a dataset
tail -f train.log        # WATCH training output live (the AI Engineer's best friend)
wc -l data.jsonl         # how many examples? (line count)
diff config_v1.yaml config_v2.yaml   # what changed between two configs
```

> [!IMPORTANT]
> **`tail -f` (follow) is the single most-used command when running AI jobs** — it streams a log file live, so you watch training loss, errors, or a server's requests in real time. Pair it with `grep`: `tail -f train.log | grep -i error` shows only errors as they happen. **Never `cat` a huge file** (it dumps everything and can flood your terminal / load a multi-GB file) — use `less` (paged, memory-friendly) or `head`/`tail`.

## 7. Searching: `find`, `locate`, `grep`

| Command | Searches for | Key forms |
|---|---|---|
| `find` | **Files** by name/type/size/time (walks the tree live) | `find . -name "*.py"`, `find /data -size +1G`, `find . -mtime -1` |
| `locate` | **Files** by name (fast, from a prebuilt index) | `locate model.safetensors` |
| `grep` | **Content** — lines matching a pattern | `grep -r "TODO" src/`, `grep -i error log`, `grep -c` (count) |

```bash
find /data -name "*.parquet" -size +100M     # large parquet files
find . -name "*.py" -mtime -1                 # Python files modified in the last day
grep -rn "api_key" .                          # find (line-numbered) where a string appears — recursively
grep -i "error\|warning" train.log            # case-insensitive, either pattern
```

| | `find` | `locate` |
|---|---|---|
| Method | Walks the filesystem now | Queries a prebuilt database (`updatedb`) |
| Freshness | Always current | Can be stale (index updated periodically) |
| Speed | Slower (real traversal) | Very fast |
| Power | Rich filters (size, time, type, `-exec`) | Name only |

> [!IMPORTANT]
> **`grep` is the most valuable text tool you'll own.** It finds matching lines in files or piped input — for searching code, filtering logs, and extracting data. Learn `grep -r` (recursive, search a codebase), `-i` (case-insensitive), `-n` (line numbers), `-c` (count), `-v` (invert — lines *not* matching), and `-E` (extended regex). `find` locates *files*; `grep` searches *content* — a common confusion. `find ... -exec` and `xargs` (§9) let you run a command on each found file.

## 8. Text Processing: `awk`, `sed`, `cut`, `sort`, `uniq`, `tr`

These are the pipeline powerhouses — small tools that transform text streams. Learn them and you can wrangle data without leaving the terminal.

| Command | Job | Example |
|---|---|---|
| `cut` | Extract columns/fields | `cut -d, -f2 data.csv` (2nd comma-separated field) |
| `sort` | Sort lines | `sort -n` (numeric), `sort -r` (reverse), `sort -k2` (by column 2) |
| `uniq` | Deduplicate / count *adjacent* duplicates | `uniq -c` (count), needs `sort` first |
| `tr` | Translate/delete characters | `tr 'A-Z' 'a-z'` (lowercase), `tr -d ' '` (delete spaces) |
| `sed` | Stream editor: find/replace, edit lines | `sed 's/old/new/g'` (substitute), `sed -n '10,20p'` (print lines) |
| `awk` | Field-based processing & computation | `awk '{sum+=$1} END{print sum}'` (sum a column) |
| `tee` | Write to file *and* pass on | `... | tee out.txt` ([03.4](03.4-terminal-mastery.md)) |

```bash
# Extract the 3rd column of a CSV, get unique values with counts:
cut -d, -f3 data.csv | sort | uniq -c | sort -rn

# Sum a numeric column with awk:
awk -F, '{sum += $2} END {print "total:", sum}' data.csv

# Replace a token across a file (stream editor):
sed 's/localhost/prod-server/g' config.txt

# Lowercase and count word frequency:
tr 'A-Z' 'a-z' < text.txt | tr -s ' ' '\n' | sort | uniq -c | sort -rn | head
```

> [!IMPORTANT]
> **`awk` and `sed` are mini-languages, not just commands.** `sed` (stream editor) excels at find-and-replace and line edits across streams/files; `awk` excels at column/field processing and arithmetic (it auto-splits lines into `$1, $2, ...`). For AI data work: `cut`/`awk` extract columns from CSV/TSV, `sort | uniq -c` counts distinct values (label distributions!), `sed` bulk-edits configs. You don't need mastery — knowing "awk for columns/math, sed for substitution" and a few idioms handles 90% of real tasks. For heavy work, reach for Python/pandas ([Module 07](../../07-Data-Analysis/README.md)); for quick inspection, these win.

> [!WARNING]
> **`uniq` only removes *adjacent* duplicates** — you must `sort` first (`sort | uniq -c`). A frequent bug is `uniq` on unsorted data missing non-adjacent dupes. Also, `sed -i` edits files **in place** (no backup unless you pass `sed -i.bak`) — a risky operation on important files; test the substitution without `-i` first.

## 9. `xargs` — Build Commands from Input

`xargs` takes items from stdin and turns them into **arguments** for another command — the bridge between "find/list things" and "do something to each."

```bash
# Find all .pyc files and delete them:
find . -name "*.pyc" | xargs rm

# Safer with null-delimiters (handles spaces in filenames):
find . -name "*.pyc" -print0 | xargs -0 rm

# Run a command on each item, in parallel (4 at a time):
cat urls.txt | xargs -P4 -n1 curl -O
```

> [!TIP]
> `xargs` shines for "do X to every file matching Y." The `-print0`/`-0` pairing handles filenames with spaces/newlines safely (a real gotcha). `xargs -P` runs commands in **parallel** — a poor-man's way to parallelize downloads or processing across files ([Module 02.8](../../02-Computer-Science/weeks/02.8-concurrency.md)). Note: `find -exec` does a similar job; `xargs` is often faster (batches arguments) and more flexible.

## 10. A Real AI Data-Wrangling Session

Putting it together — inspecting and cleaning a dataset entirely from the terminal:

```bash
wc -l data.jsonl                                    # how many examples?
head -1 data.jsonl | python -m json.tool            # inspect one example's structure
cut -d, -f5 labels.csv | sort | uniq -c | sort -rn  # label distribution (class balance!)
grep -c '"label": null' data.jsonl                  # count records with null labels
find /data -name "*.tmp" -mtime +7 | xargs rm       # clean old temp files
tail -f training.log | grep -iE "loss|error"        # live-monitor training
```

Every one of these is faster than opening a notebook — the terminal as a data tool.

## 11. Common Mistakes & Debugging

| Mistake | Consequence | Fix |
|---|---|---|
| `rm -rf` on wrong/empty path | Data loss | `ls` first; `"${DIR:?}"`; trash instead |
| `cat` a huge file | Terminal flooded / slow | `less`, `head`, `tail` |
| `uniq` without `sort` | Missed non-adjacent dupes | `sort | uniq` |
| `sed -i` without backup | Overwrote a file | Test without `-i`; use `-i.bak` |
| `find`/`xargs` with spaces in names | Broken/wrong targets | `-print0 | xargs -0` |
| Confusing `find` (files) vs `grep` (content) | Wrong tool | `find`=files, `grep`=text inside |
| Forgetting `-r` on `grep`/`cp` | Only top level | Add `-r` for recursion |

## 12. Performance Considerations

| Principle | Takeaway |
|---|---|
| `less`/`tail` don't load whole files | Essential for multi-GB logs/data |
| Pipes stream | Process data larger than RAM ([03.4](03.4-terminal-mastery.md)) |
| `grep`/`awk` are fast (C) | Often beat naive Python for line filtering |
| `locate` vs `find` | Index lookup vs live traversal |
| `xargs -P` | Parallelize file operations |
| Right tool at scale | For heavy transforms, Python/pandas ([Module 07](../../07-Data-Analysis/README.md)) |

## 13. Security Considerations

| Risk | Guidance |
|---|---|
| `grep` for secrets accidentally logged | Also a *use*: `grep -rn "API_KEY" .` to audit for leaked secrets |
| Destructive commands from untrusted input | Never build `rm`/`sed` commands from untrusted strings (injection, [03.4](03.4-terminal-mastery.md)) |
| Reading sensitive files | Respect permissions ([03.6](03.6-permissions.md)); don't world-expose |
| `find -exec`/`xargs` on untrusted trees | Malicious filenames/symlinks — use `-print0`, validate |

> [!TIP]
> Turn these tools on security *for* you: `grep -rn "api_key\|password\|secret" .` scans a codebase for accidentally-committed secrets (a real pre-commit check), and `find / -perm -4000` lists SUID binaries ([03.6](03.6-permissions.md)). The same tools that wrangle data audit for problems.

## 16. Cheat Sheet

```text
NAV: pwd · ls -lh(sizes)/-la(hidden)/-lt(time) · cd ~ / cd - / cd .. · tree -L 2
FILES: cp -r · mv old new · mkdir -p a/b/c · touch f · ln -s target link
  rm file · rm -r dir · rm -rf ⚠️IRREVERSIBLE (ls first! guard "${DIR:?}"; no trash bin)
VIEW: cat(small) · less(big, paged) · head -n / tail -n · TAIL -F (live log!) · wc -l · diff a b
SEARCH: find . -name "*.py" -size +1G -mtime -1   |  locate name(fast index)  |  grep
  grep -r(recursive) -i(ignore case) -n(line#) -c(count) -v(invert) -E(regex)
TEXT: cut -d, -f2 · sort -n/-r/-k2 · uniq -c (SORT FIRST!) · tr 'A-Z' 'a-z' · sed 's/old/new/g' · awk -F, '{s+=$2}END{print s}'
  tee file(save+pass) · xargs (stdin→args; -print0|xargs -0 for spaces; -P parallel)
IDIOMS: cut -d, -fN | sort | uniq -c | sort -rn   (value counts / label distribution)
        tail -f log | grep -iE "loss|error"        (live monitor)
DANGER: rm -rf empty var · cat huge file · uniq unsorted · sed -i no backup
```

## 17. Flashcards

- **Q:** `find` vs `grep`? — **A:** `find` searches for *files* (by name/size/time, walking the tree); `grep` searches *content* (lines matching a pattern) in files or piped input.
- **Q:** Why is `tail -f` so important for AI work? — **A:** It follows a log live, streaming new lines as they're written — how you watch training loss, errors, or a server in real time.
- **Q:** Why must you `sort` before `uniq -c`? — **A:** `uniq` only collapses *adjacent* duplicates, so unsorted data leaves non-adjacent dupes uncounted.
- **Q:** `awk` vs `sed`? — **A:** `awk` for column/field processing and arithmetic (auto-splits into `$1,$2,...`); `sed` for stream find/replace and line edits.
- **Q:** How do you get a column's value counts (e.g., label distribution)? — **A:** `cut -d, -fN file | sort | uniq -c | sort -rn`.
- **Q:** Why is `rm` especially dangerous? — **A:** It's permanent (no trash); `rm -rf` on a wrong or empty-variable path can wipe data/systems — always `ls` first and guard variables.
