# 17.8 · Containers ✅

[⬅ 17.7 Databases for AI Systems](17.7-databases.md) · [🏠 Module 17](../README.md) · [➡ 17.9 Kubernetes for AI Engineers](17.9-kubernetes.md)

> **The lesson in one line:** A container packages your application **with its exact dependencies, libraries, and runtime** into a portable image that runs identically on your laptop, a CI runner, and a cloud GPU node — killing "works on my machine." For AI, where CUDA versions, Python packages, and model runtimes are notoriously fragile, containers are the unit of reproducible deployment: **build an image once, run it anywhere.**

## 🧠 Mental model

> [!IMPORTANT]
> **A container is a lightweight, isolated box that holds your app *and everything it needs to run* — so the environment travels with the code.** A VM virtualizes the *hardware* (each VM carries a full OS — heavy, slow to boot). A container virtualizes the *operating system* — it shares the host kernel but isolates its own filesystem, processes, and dependencies (lightweight, boots in milliseconds, packs densely). The key artifacts: a **Dockerfile** is the recipe; building it produces an **image** (an immutable, layered snapshot of your app + deps); running an image produces a **container** (a live instance); images are shared through a **registry**. For AI this solves a real pain: your model needs a specific CUDA + PyTorch + library combination, and a container pins all of it so the GPU node runs exactly what you tested.

## 🔍 Internal explanation

### Containers vs. VMs

| | VM | Container |
|---|---|---|
| Virtualizes | hardware (own OS per VM) | the OS (shares host kernel) |
| Size | GBs | MBs–small GBs |
| Boot | seconds–minutes | milliseconds |
| Isolation | strong (separate kernel) | process-level (shared kernel) |
| Density | fewer per host | many per host |
| AI use | GPU host, full control | the app/model unit you ship & scale |

Containers and VMs compose: cloud GPU **VMs** run a container **runtime**, and Kubernetes ([17.9](17.9-kubernetes.md)) schedules **containers** onto those VMs.

### The core artifacts

- **Dockerfile** — a text recipe: base image, copy code, install deps, set the start command. Deterministic and version-controlled ([16.21](../../16-MLOps/weeks/16.21-iac.md)).
- **Image** — the built result: an immutable, **layered** snapshot. Layers are cached and shared, so rebuilding after a code change only rebuilds the top layers (fast).
- **Container** — a running instance of an image; ephemeral by default (its filesystem vanishes when it stops — hence volumes).
- **Registry** — where images are stored and pulled from (Docker Hub, or cloud registries: ECR/ACR/Artifact Registry). CI builds an image and pushes it; the cloud pulls it to run.

> [!IMPORTANT]
> **Images are immutable and reproducible; containers are ephemeral and disposable.** This is the whole philosophy: you build a versioned image (`myapp:1.4.0`) that is byte-identical everywhere, and you run many throwaway containers from it. If a container misbehaves, you kill it and start a fresh one — no snowflake servers, no manual fixes that get lost. This is what makes containers the unit of **reproducible, scalable** deployment ([16.2](../../16-MLOps/weeks/16.2-reproducibility.md)).

### Networking and volumes

- **Container networking** — containers get their own network namespace; you **publish ports** to expose a service (e.g. map container `:8000` to the host), and containers on a network reach each other by name. In the cloud this integrates with the VPC ([17.5](17.5-networking.md)).
- **Volumes** — because a container's filesystem is ephemeral, **volumes** persist data outside the container lifecycle (a mounted disk or object-storage-backed path). Model weights and data usually come from a mounted volume or are pulled from object storage at startup, not baked into the image if large.

### The AI-specific concerns

- **Base image** — for GPU workloads, start from a CUDA-enabled base so the container can use the GPU (the host also needs the GPU runtime).
- **Image size** — AI images balloon (CUDA + PyTorch + libs can be many GB); use slim bases, multi-stage builds, and **don't bake huge model weights into the image** (pull from object storage/registry at runtime — [17.6](17.6-storage.md), [16.5](../../16-MLOps/weeks/16.5-model-registry.md)).
- **Pinned dependencies** — pin versions (lock files) so the image is reproducible ([16.2](../../16-MLOps/weeks/16.2-reproducibility.md)).

## 🛠️ Practical implementation

```dockerfile
# A model-serving image — the AI container pattern
FROM nvidia/cuda:12.1-runtime-ubuntu22.04        # GPU-capable base
WORKDIR /app
COPY requirements.lock .                          # pinned deps → reproducible (16.2)
RUN pip install --no-cache-dir -r requirements.lock
COPY app/ /app                                    # your FastAPI/serving code
# NOTE: large model weights are pulled from object storage at startup, NOT baked in
EXPOSE 8000
CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
```
```bash
docker build -t myorg/llm-api:1.0.0 .        # build the image (layered, cached)
docker run --gpus all -p 8000:8000 \         # run a container, expose GPU + port
  -e MODEL_URI=s3://models/llm/v1 myorg/llm-api:1.0.0
docker push myorg/llm-api:1.0.0              # push to a registry for the cloud to pull
```

**The deployment flow:** commit code → CI builds & tests the image → push to registry → cloud (K8s/serverless/VM) pulls and runs it → monitor ([17.17](17.17-deployment.md)).

## ⚡ Performance considerations

- **Layer caching** — order the Dockerfile so rarely-changing steps (deps) come before frequently-changing ones (code); rebuilds stay fast.
- **Image size affects cold start** — big images take longer to pull onto a new node; keep them lean (multi-stage, slim base), especially for autoscaling ([17.15](17.15-autoscaling.md)).
- **Don't bake giant weights in** — a 30 GB image is slow to pull everywhere; pull weights from object storage at startup.

## 💲 Cost considerations

- **Registry storage & pulls** — many large image versions cost storage and transfer; prune old tags.
- **Smaller images pull faster** → faster autoscaling → less over-provisioning ([17.14](17.14-cost-optimization.md)).
- **Reproducible images prevent costly "rebuild to debug"** cycles.

## 🔒 Security considerations

> [!CAUTION]
> - **Scan images for vulnerabilities** and use **minimal, pinned** base images (smaller attack surface) ([17.13](17.13-security.md)).
> - **Never bake secrets into images** — images are shared via registries; pass secrets at runtime from a secrets manager ([17.13](17.13-security.md)).
> - **Run as non-root** inside the container; drop unneeded capabilities.
> - **Trust your base images** — pin by digest; a compromised base is a supply-chain risk.
> - **Private registries** for proprietary models/code.

## 🚫 Common mistakes

| Mistake | Consequence |
|---|---|
| Secrets baked into the image | leaked via the registry |
| Baking huge model weights into the image | slow pulls, bloated registry |
| Unpinned deps / `latest` base tags | non-reproducible builds ([16.2](../../16-MLOps/weeks/16.2-reproducibility.md)) |
| Running as root | larger blast radius on compromise |
| Poor layer ordering | slow rebuilds |
| Treating containers as pets (manual fixes) | snowflake, unreproducible state |

## 🐛 Debugging workflow

Container issue: (1) **"Works locally, fails in cloud."** → Usually a base-image/GPU-runtime or missing-env-var difference; confirm the same image tag runs both places and the GPU runtime is present. (2) **Container won't start.** → Check the start command, missing env/secrets, and whether the port is right; read the container logs. (3) **GPU not visible in container.** → Missing `--gpus`/device plugin or a non-CUDA base image ([17.4](17.4-gpu-infrastructure.md)). (4) **Image huge / slow pulls.** → Multi-stage build, slim base, weights out of the image. (5) **Data lost on restart.** → Ephemeral filesystem — mount a volume or use object storage.

## 🛠️ Mini project — "Containerize an AI service" ✅

**Goal:** a production-quality container for an LLM (or RAG) API.

**Requirements:** a Dockerfile with a GPU-capable base, pinned dependencies, non-root user, and **no baked-in secrets or weights** (pull weights from object storage at startup, inject secrets at runtime); build and push to a registry; run it exposing the GPU and a port; a `.dockerignore` and multi-stage build to keep it lean; an image scan step.
**Folder structure**
```
llm-api/
├── Dockerfile          # GPU base, pinned deps, non-root
├── .dockerignore
├── requirements.lock   # pinned (16.2)
├── app/                # serving code
└── Makefile            # build / scan / push / run
```
**Testing:** the image runs identically locally and on a cloud GPU node; GPU is visible; no secrets in the image.
**Security:** scanned, non-root, pinned base by digest ([17.13](17.13-security.md)). **Monitoring:** container logs + health endpoint.
**Future improvements:** multi-stage size reduction; add to a CI/CD pipeline ([17.17](17.17-deployment.md)); deploy to Kubernetes ([17.9](17.9-kubernetes.md)).

## 🎴 Flashcards

- **⭐ What is a container and what problem does it solve?** → An app packaged with its exact dependencies/runtime in a portable, OS-virtualized box — it kills "works on my machine" by making the environment travel with the code.
- **Container vs. VM?** → A VM virtualizes hardware (full OS each, heavy); a container virtualizes the OS (shares the host kernel, lightweight, ms boot, dense).
- **Image vs. container?** → An image is the immutable layered snapshot; a container is a running, ephemeral instance of it.
- **⭐ The build-to-deploy flow?** → Source → Dockerfile → Image → Registry → Container → Cloud.
- **Why not bake large model weights into the image?** → Bloated images pull slowly everywhere (hurting autoscaling); pull weights from object storage at startup instead.
- **Why never bake secrets into an image?** → Images are shared via registries; inject secrets at runtime from a secrets manager.
- **What's a volume for?** → Persisting data outside the container's ephemeral filesystem.
- **Why order Dockerfile steps deps-before-code?** → Layer caching — a code change then rebuilds only top layers, keeping builds fast.
- **What makes a container GPU-capable?** → A CUDA-enabled base image plus the host GPU runtime/device exposure.

## 📝 Summary

- A **container** packages an app with its exact dependencies into a portable, OS-virtualized box — solving the reproducibility problem that plagues AI (CUDA/PyTorch/library version hell).
- The artifacts: **Dockerfile** (recipe) → **image** (immutable, layered, reproducible) → **registry** (share) → **container** (ephemeral running instance) → cloud; **images are immutable, containers are disposable**.
- For AI: use a **CUDA base**, **pin dependencies**, keep images lean, and **pull weights from object storage and secrets from a manager at runtime** — never bake them in ([17.6](17.6-storage.md), [17.13](17.13-security.md)).
- Containers are the **unit of deployment and scaling** — the next lesson orchestrates many of them with **Kubernetes** ([17.9](17.9-kubernetes.md)), and the deployment pipeline that builds/ships them is [17.17](17.17-deployment.md).
